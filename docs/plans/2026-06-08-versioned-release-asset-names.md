# Versioned Release Archive Names Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task after user approval.

**Status:** Approved by user for current `v0.3.0` release mutation. User clarified: version the archive filenames only; do **not** publish separate standalone `bn.exe` or `sql-wasm.wasm` release assets.

**Goal:** Add the release version to BlueNote's public portable archive filenames so downloaded archives are self-describing and harder to mix across releases.

**Architecture:** Keep the executable names and SQL.js companion file inside portable packages unchanged (`bn.exe`/`bn` and `sql-wasm.wasm`). Publish only two GitHub Release assets for each release: one versioned Windows ZIP and one versioned Linux tarball. Derive the asset version from the Git tag when available, with a local/dev override or `package.json` fallback, and make packaging, workflow upload/download, checksums, release docs, and tests agree on the same contract.

**Tech Stack:** Bun 1.3+, TypeScript release packaging script, GitHub Actions release workflow, `gh` for release verification.

**Public asset names for `v0.3.0`:**

- `bluenote-v0.3.0-windows-x64.zip`
- `bluenote-v0.3.0-linux-x64.tar.gz`

**No separate public assets:**

- No standalone `bn.exe`
- No standalone `sql-wasm.wasm`

**Internal package layout stays:**

```text
bluenote/
  bn.exe        # Windows ZIP
  bn            # Linux tarball
  sql-wasm.wasm
  README.txt
```

## Task 2: Version archive names in `scripts/package-release.ts`

**Objective:** Make local packaging produce versioned public archive filenames while preserving the portable archive's internal layout and removing standalone public output files.

**Files:**
- Modify: `scripts/package-release.ts`

**Steps:**
1. Add a helper to derive release version in this order:
   - `BLUENOTE_RELEASE_VERSION` if set
   - `GITHUB_REF_NAME` if it starts with `v`
   - `v${packageJson.version}` fallback
2. Change platform archive names to include the derived version:
   - Windows ZIP: `bluenote-${version}-windows-x64.zip`
   - Linux tarball: `bluenote-${version}-linux-x64.tar.gz`
3. Keep `release.executableName` unchanged inside archives (`bn.exe`/`bn`).
4. Remove standalone executable/standalone WASM publishing from the package script; `sql-wasm.wasm` should exist only inside the archive package root.
5. Update package validation to check the versioned archive path exists and still contains internal `sql-wasm.wasm`.
6. Run:

```bash
BLUENOTE_RELEASE_VERSION=v0.3.0 bun run package:release
```

Expected: `dist/release/bluenote-v0.3.0-linux-x64.tar.gz` exists locally and extraction contains `bluenote/bn`, `bluenote/sql-wasm.wasm`, and `bluenote/README.txt`.

## Task 3: Update the GitHub release workflow

**Objective:** Upload, checksum, and publish only the two versioned archives generated by packaging.

**Files:**
- Modify: `.github/workflows/release.yml`

**Steps:**
1. Pass the tag name to packaging on release/tag builds:

```yaml
env:
  BLUENOTE_RELEASE_VERSION: ${{ github.ref_name }}
```

2. Update `actions/upload-artifact` paths:
   - Windows artifact uploads `dist/release/bluenote-${{ github.ref_name }}-windows-x64.zip`
   - Linux artifact uploads `dist/release/bluenote-${{ github.ref_name }}-linux-x64.tar.gz`
3. Remove direct upload of `dist/release/bn.exe` and `dist/release/sql-wasm.wasm`.
4. Update checksum generation to include only the two archive filenames.
5. Update `softprops/action-gh-release` `files:` to publish only the two archive filenames.
6. Keep artifact directory names (`bluenote-windows-x64`, `bluenote-linux-x64`) stable.

- Only archive assets are published; no separate `bn.exe` or `sql-wasm.wasm` assets.
- Portable archive internals stay simple: `bn.exe`/`bn`, `sql-wasm.wasm`, `README.txt`.
- Workflow upload paths, checksum generation, and release publishing cannot drift.
- Docs explain versioned archive downloads and unchanged internal executable names.
- Local packaging and full public gate pass.
- Current `v0.3.0` is republished and verified from fresh GitHub downloads.
